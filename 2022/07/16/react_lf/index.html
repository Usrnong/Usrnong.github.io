react 新旧两版生命周期需要注意的内容（以16.8分界） | react 16.8 +

发表于2022-07-16|更新于2023-07-14|react教程

componentWillMount

  • 16.2版本之后 如果坚持使用，则添加前缀UNSAFE_componentWillMount
  • react diff算法更新，提出fieber技术。在发现新的状态更新之后，会同步对比新老dom树的，当组件多了之后极有可能会造成浏览器的假死现象，fieber技术优化了虚拟dom的diff算法，把创建dom和组件渲染过程拆分成多个分片任务来执行。任务碎片化，如果有优先级较高的任务就执行较高的任务，如果有优先级高的任务插入会打断低优先级的任务，所以componentWillMount极可能会被打断。重复操作可能会多次触发，失去了唯一性，存在隐患。可以使用新的生命周期或者把操作迁移到componentDidMount

componentWillReceiveProps(nextProps)

  • 组件中修改状态不会执行，但是只要父组件中的状态修改，就会触发组件的生命周期，即使没传属性，此组件也回执行更新流程，触发此componentWillReceiveProps生命周期，直接读取this.props属性会得到旧的属性，参数nextProps会最先获得父组件传过来的属性值，此时可以进行逻辑操作或者ajax或者把属性转化为自己的状态。
  • 外部组件多次频繁更新传入多次不同的 props，会导致不必要的异步请求
  • 在子组件中写这个生命周期才有意义。

componentWillUpdate

  • 组件即将更新，不能修改属性和状态，dom还没更新，可以获取更新之前的dom结构
  • 更新前记录 DOM 状态, 可能会做一些处理，与componentDidUpdate相隔间如果过长，会导致状态不可信

getDerivedStateFromProps(nextProps, nextState)

  • 第一次的初始化组件以及后续的更新过程中都会执行(包括自身状态更新以及父传子) ，返回一个对象作为新的state，返回null则说明不需要在这里更新state
  • 使用这个生命周期时组件中必须含有已经定义好的状态对象，并且生命周期函数中必须返回一个对象作为新的state；并且这个生命周期是类当中的静态属性static getDerivedStateFromProps。
  • 返回的对象，最后会与状态对象合并。
  • 适用于初始化和后续更新中都需要重复操作的逻辑。
  • 此生命周期作为新的生命周期可以取代不安全的 componentWillMount componentWillReceiveProps
  • 无法在此生命周期中无法进行异步的数据请求，函数中会直接立即执行return 不会等待异步返回。在官方建议中提到，在此函数中只是做一些属性转换状态或者其他的属性状态同步操作。再配合使用componentDidMount和componentDidUpdate可完成异步请求。
  • 因为getDerivedStateFromProps在一次事件循环中多次处理，到最后也会将操作合并成为一次return到新的状态中，不用担心多次更新带来的异步操作。所以到componentDidMount和componentDidUpdate中就可以获取最新的状态

getSnapshotBeforeUpdate(prevProps, prevState)

  • 取代了 componetWillUpdate ,触发时间为update发生的时候，在render之后dom渲染之前返回一个值，作为componentDidUpdate的第三个参数
